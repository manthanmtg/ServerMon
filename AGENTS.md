This file defines rules and expectations for any automated agent, CI pipeline, or bot that operates on this repository.

## Pre-merge Requirements

Every pull request and every commit to `main` must satisfy:

| Check  | Command          | Expectation          |
| ------ | ---------------- | -------------------- |
| Format | `pnpm format`    | 0 changes needed     |
| Lint   | `pnpm lint`      | 0 errors, 0 warnings |
| Types  | `pnpm typecheck` | 0 errors             |
| Build  | `pnpm build`     | Exit code 0          |
| Tests  | `pnpm test`      | Exit code 0          |

Combined: `pnpm check` runs all of the above in order. **A failure in any step blocks the merge.**

## Repository Health Rules

### No Ignored Errors

- Do not add `eslint-disable` comments without a paired explanation comment.
- Do not add `@ts-ignore` or `@ts-expect-error`. Fix the type issue instead.
- Do not add `any` types. Use `unknown` and narrow with type guards.

### No Dead Code

- Remove unused imports, variables, functions, and files.
- Do not comment out code and commit it. Use git history.

### No Secrets

- Never commit `.env`, `.env.local`, credentials, API keys, or tokens.
- `.env.example` is the only env file that should be tracked.
- JWT secrets are auto-generated by the installer. Never hardcode them.

### No Large Files

- `.pnpm-store/`, `node_modules/`, `.next/`, and build artifacts must never be committed.
- These are in `.gitignore`. If git tries to track them, something is wrong.

### Dependency Hygiene

- Use `pnpm` exclusively. Do not use npm or yarn.
- Do not add dependencies without a clear need.
- `@types/*` packages for libraries that ship their own types are unnecessary (e.g., `@types/mongoose` â€” mongoose has built-in types).
- Pin major versions in `package.json`. Use `^` for minor/patch.

## File Conventions

### Naming

| Type       | Convention                           | Example                          |
| ---------- | ------------------------------------ | -------------------------------- |
| Pages      | `page.tsx` in route folder           | `src/app/dashboard/page.tsx`     |
| API routes | `route.ts` in API folder             | `src/app/api/health/route.ts`    |
| Components | PascalCase `.tsx`                    | `ProShell.tsx`, `Button.tsx`     |
| Utilities  | camelCase `.ts`                      | `utils.ts`, `logger.ts`          |
| Types      | PascalCase interface, camelCase file | `module.ts` â†’ `Module` interface |
| Modules    | lowercase folder name                | `src/modules/terminal/`          |

### New Component Checklist

When creating a new UI component in `src/components/ui/`:

1. Use `forwardRef` for elements that accept refs.
2. Accept a `className` prop and merge with `cn()`.
3. Use semantic theme tokens â€” never hardcode colors.
4. Export the component and its props type.
5. Ensure minimum 44px touch target for interactive elements.

### New Module Checklist

When creating a new module in `src/modules/<name>/`:

1. Create `module.ts` with the `Module` interface from `src/types/module.ts`.
2. Create UI in `ui/` subfolder â€” widget and/or page components.
3. Register the widget in `src/components/modules/ModuleWidgetRegistry.tsx`.
4. Add the nav entry in `src/components/layout/ProShell.tsx` `navGroups`.
5. Create the page route in `src/app/<name>/page.tsx` wrapped in `<ProShell>`.
6. All API routes go in `src/app/api/modules/<name>/`.
7. Use `createLogger('<name>')` for logging.
8. Wrap widgets in `WidgetErrorBoundary`.
9. Run `pnpm check` to verify.

### New API Route Checklist

1. Wrap the handler body in try/catch.
2. Use `createLogger('api:<name>')`.
3. Return `{ error: "message" }` on failures with proper HTTP status.
4. Add `export const dynamic = 'force-dynamic'` if accessing DB or runtime state.
5. Cap any user-provided `limit` parameters.
6. Validate input with Zod if accepting a request body.
7. **Security**: Ensure all routes require authentication via `getSession()`. Only whitelisted public routes in `src/middleware.ts` should be unauthenticated.
8. **Authorization**: Implement role-based checks (`admin` vs `user`) for management or destructive operations.

## Testing Guidelines

Tests are implemented using **Vitest** for unit/integration tests and **Playwright** for E2E tests.

- **Unit Tests**: Place tests next to the file they test or in a `__tests__` folder: `file.test.ts` or `ui/Component.test.tsx`.
- **Runner**: Use `vitest`.
- **Execution**:
  - `pnpm test`: Run all unit tests once (included in `pnpm check`).
  - `pnpm test:watch`: Run unit tests in watch mode.
  - `pnpm test:ui`: Open Vitest UI.
  - `pnpm test:coverage`: Generate coverage report.
  - `pnpm test:e2e`: Run Playwright E2E tests (optional, manual only).
- **Mocking**: Use `vi.mock()` or `vitest-mock-extended` for dependency injection.
- **Coverage**: Aim for high coverage on core utility logic and API handlers.

| ------------ | -------- | -------------------------------------------------- |
| `MONGO_URI`  | Yes      | MongoDB connection string                          |
| `JWT_SECRET` | Yes      | Secret for signing session JWTs                    |
| `PORT`       | No       | App port (default: 8912)                           |
| `NODE_ENV`   | No       | `development` or `production`                      |
| `LOG_LEVEL`  | No       | `debug`, `info`, `warn`, `error` (default: `info`) |

## Architecture Decisions

These are intentional choices. Do not change them without discussion:

- **Single SSE connection pattern**: `MetricsProvider` creates one EventSource. Widgets consume via `useMetrics()`. Do not create EventSource instances in components.
- **CSS variable theming**: All colors flow through CSS variables defined in `globals.css` and applied via Tailwind's `@theme inline`. This is what makes the 6-theme system work.
- **Error boundaries per widget**: `ModuleWidgetRegistry` wraps each widget so one crashing module doesn't take down the page.
- **Middleware protects all routes**: The default is authenticated. Public routes are an explicit allowlist in `src/middleware.ts`.
- **Structured logging**: `createLogger()` outputs `TIMESTAMP [LEVEL] [CONTEXT] Message` to stdout, which systemd captures into journald.
- **Idempotent installer**: `scripts/install.sh` can be run multiple times safely. It preserves secrets on re-run.
